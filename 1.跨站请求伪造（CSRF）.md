# 1.跨站请求伪造（CSRF）

1.定义：CSRF 就是：利用浏览器自动带 Cookie 的特点，骗用户在已登录状态下，自动发出恶意请求，替攻击者干坏事。

它只改状态，不偷数据。（只要对方提交上了cookie，并且也能带走我们的一些参数或者数据，只要我们的参数或者数据能被服务器识别，那么我们就成功了）

## 1.无效的预防措施

### 1.使用 “秘密 Cookie” → 无效

你就算把 Cookie 设得再复杂、再保密，也没用。

原因：

- 浏览器**不管是不是恶意请求**，只要访问目标网站，就会**自动带上所有 Cookie**。
- Cookie 只能证明**你是谁**，不能证明**这个请求是你自愿发的**。

结论：

**Cookie 再机密，也防不住 CSRF。**

### 2.只接受 POST 请求 → 无效

很多人以为：

只用 POST，不用 GET，攻击者就没法构造链接 = 安全。

**错得离谱。**

攻击者可以：

- 在恶意页面写一个**隐藏表单**
- 用 JS **自动提交 POST**
- 你一打开页面，直接发请求

结论：

**POST 根本防不住 CSRF。**

### 3.多步交易（比如分两步改密码）→ 无效

有人觉得：操作分好几步，攻击者就伪造不了。

错。

只要步骤是**固定、可预测**的，攻击者就能把整个流程拼成一个请求。

结论：

**多步操作 ≠ 防 CSRF。**

### 4.URL 重写（把 SessionID 放 URL 里）→ 无效 + 更危险

有人想：

把 SessionID 放 URL 里，攻击者猜不到，就安全。

**大错特错，还引入新漏洞：**

- SessionID 直接暴露在 URL 里
- 复制链接、分享、历史记录都会泄露
- 直接导致**会话劫持**

原文说的很对：

**不要用一个漏洞去修另一个漏洞。**

### 5.HTTPS → 不能防 CSRF

HTTPS 只防：

- 窃听
- 中间人篡改

但 HTTPS **不改变**浏览器的行为：

**该自动带的 Cookie 照样自动带。**

结论：

**HTTPS 是基础安全，但不防 CSRF。**

### 6.验证 Referer（来源头）→ 不靠谱

很多新手喜欢用：

判断请求是不是从自己网站来的。

但这东西：

- 能被**伪造**
- 浏览器可以**禁用 Referer**
- 从书签、本地文件打开时 Referer 为空 → 误杀正常用户

结论：

**Referer 只能当辅助，不能当主防御。**

## 2.真实攻击的完整流程（简化版）

**侦察**：你登录目标网站，抓包分析修改密码 / 转账的请求参数和地址。

（为后面的写代码做准备，你要往里面写代码，你最起码得知道把代码写在哪里，对吧？把代码写成什么样子？）

**构造 payload**：根据 GET/POST 类型，写好恶意 HTML 页面。

（嗯，造料子，你要往里面加料，你最起码得有料子，通过刚才的观察，你要知道自己要往里面写什么？）

**投放**：把恶意页面上传到你的服务器，生成短链接，通过邮件 / 社交平台发给受害者。

（欺骗受害者，不管是，跳转也好，访问也罢，总之，要让他从我们这过，只要它从我们这里过了，我们就可以往里面加料，但是至于这个量能不能加进去，就看你的水平了）

**触发**：受害者在已登录目标网站之前的状态下，点开链接→自动执行恶意请求。

（吃料子，对方已经在不知情的情况下，把我们的代码给服务器交上去了，就相当于对方把我们的料子吃下去了）

**得手**：服务器验证 Cookie 合法，执行操作（比如改密码、转账），你就拿到了账号权限 / 资金。

注意注意：写好恶意 HTML 页面。是让你登录用的。

## 3.示例（经典的吃料子模型）

1(侦察).打开DVWA，然后观察它那个改密码的漏洞，它是从哪提交的？从URL提交就做URL的HTML，从post提交就做post的HTML

```php
if( isset( $_GET[ 'Change' ] ) ) {
    // Get input
    $pass_new  = $_GET[ 'password_new' ];
    $pass_conf = $_GET[ 'password_conf' ];

```

由代码可知啊，这个提交代码的地方是URL，应该提交get

```
password_new=password&password_conf=password&Change=变化#
```

它的URL里格式是这样的

2.构造 payload:第二步自己造料子

```php
<a href="http://localhost/dvwa/vulnerabilities/csrf/password_new=hacker123&password_conf=hacker123&Change=Change">
        点击领取福利
    </a>
```

这个a跳转，当a跳转的时候，它会到达DVWA，因为你的浏览器中有cookie你点击登录的时候，它就会自动登录成功，但是因为你是经过我这个跳转程序到的DVWA，所以我改变密码的代码也会跟着你进入DVWA

3.投放:放料子，我可以通过邮件乱七八糟的附件诱导你点我的文件

4.**触发**：吃料子，当你进入DVWA中以后，如果也去更新密码，当你提交请求的那一刻，我写好的代码也会跟着你的请求一起提交，

5.成功：这样我既不需要知道你的cookie，也不需要知道它的防御系统是什么，我只知道，它提交表单的格式，并且我能诱导你点击我的文件，能把我的文件夹在你的运行代码当中，我就成功了

## 4.防御手段

### 1.中级防御手段

1.代码

```php
stripos( $_SERVER[ 'HTTP_REFERER' ] ,$_SERVER[ 'SERVER_NAME' ]) !== false
```

2.$_SERVER[ 'HTTP_REFERER' ]：访问服务器可以有很多窗口，嗯，这个变量就是告诉服务器，我是从哪个窗口进来的？准确的说，我是在访问哪个页面，它来源于你请求表的请求头

3.$_SERVER[ 'SERVER_NAME' ]：表示本地，这个来源于服务器的配置参数，嗯，主要是标明你在哪块区域？如果你是本地服务器，嗯，则表示本地

4.代码防御逻辑：拿你的本地标识符和访问服务器的路径做对比，如果你是本电脑访问的服务器，那么你的路径里一定有本地标识符，如果没有认明你是从其他电脑经过这个电脑访问的服务器，那么服务器将不会为你服务

5.绕过：想办法知道$_SERVER[ 'SERVER_NAME' ]，然后把它写在你的请求里就可以了，这样请求投$_SERVER[ 'HTTP_REFERER' ]，和$_SERVER[ 'SERVER_NAME' ]对比，就不会产生错误

### 2.高级防御手段

1.代码

```
checkToken( $token, $_SESSION['session_token'] );
```

2.检查$token（详细见JavaScript攻击的笔记）

3.绕过

**必须配合 XSS**：纯 CSRF 无法绕过，必须先拿到用户当前有效的 Token。

- 用 XSS 窃取当前页面的 `user_token`
- 立即用这个 Token 发起修改密码请求
- 因为 Token 是一次性的，必须在页面刷新前完成攻击
